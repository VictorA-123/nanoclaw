## Architecture
nanoclaw.service (node, dist/index.js)
├── Router — maps WhatsApp group messages to the right agent
├── IPC — coordinates orchestrator and agent containers
├── Container runner — spawns ephemeral Docker containers
├── Task scheduler — Claude-driven scheduled tasks per group
│
├── groups/
│   ├── whatsapp_main/       Jammer
│   └── whatsapp_builder/    Haddock
│       └── victor-bot/      Victor's static recruiter bot (nginx served)
│
├── store/               NanoClaw internal state — do not write externally
│   ├── messages.db      SQLite: chats, messages, scheduled_tasks, sessions, registered_groups
│   └── auth/            WhatsApp session credentials — never delete
│
└── data/
├── env/                 environment per container
├── ipc/                 IPC sockets
└── sessions/            per-group Claude session data — never delete

### Group folder permissions — 777 is intentional, do not "fix"

`groups/whatsapp_main` and `groups/whatsapp_builder` are `drwxrwxrwx` (777) by
design. The service runs as root (uid 0) but agent containers run as the `node`
user (uid 1000, root:root-owned bind mount). `container-runner.ts` re-applies
`chmod 0o777` to the group folder on **every** container spawn so uid 1000 can
write its workspace. Tightening to 750 breaks agent workspace access AND is
silently reverted on the next run. Only active/registered groups reach that code
path, which is why inactive group folders remain 755. Changing this requires a
code change (chown to node:node + 750) + rebuild + restart — not a manual chmod.

NanoClaw does not know about VAN Dashboard.

POST http://localhost:4000/api/internal/token-usage

### NanoClaw ← VAN Dashboard (Phase 3)
VAN Dashboard writes /opt/van-dashboard/data/network_config.json.
NanoClaw watches this file with fs.watch() and reloads agent config on change.
Do NOT restart WhatsApp sessions on config reload — only reload agent definitions.

## Adding a new agent (current process)

1. Create /root/nanoclaw/groups/whatsapp_newagent/
2. Add CLAUDE.md (memory file) there
3. Add context to /opt/van-dashboard/data/contexts/newagent.md
4. Edit NanoClaw router to map the new WhatsApp group JID
5. npm run build && systemctl restart nanoclaw

Phase 3 target: this whole flow happens from VAN Dashboard UI. No SSH needed.

## Known issues

### VAN Dashboard context editor and auto-regen target different files (Option B follow-up)
The dashboard has two parallel context-write paths that disagree:

- `lib/contextFiles.js::regenerateAgentContext` writes `<group>/dashboard-context.md` on every memory mutation. This is what Jammer/Haddock/Sketch now load via `@dashboard-context.md` in their `CLAUDE.md`. Format: `### Title _(Cat, tier)_` h3 blocks.
- `routes/contexts.js` GET/PUT (the dashboard's context editor UI) reads `network_config.json`'s `context_file` field, which points at the dead `/opt/van-dashboard/data/contexts/{agent}.md`. Format: `- **Title** (Cat, tier) — desc` bullets. The two formats are incompatible.

Fixes needed:
1. Update `routes/contexts.js` parser + serializer to match the `### Title _(...)_` format that `regenerateAgentContext` produces.
2. Update `network_config.json` `context_file` for each agent to point at `<group>/dashboard-context.md`.
3. Decommission `/opt/van-dashboard/data/contexts/` — no code reads from it once (1) and (2) are done.

Until this is fixed, do not edit agent context from the dashboard UI — the changes write to a file no agent reads. Edit memory items in the DB instead; `regenerateAgentContext` will pick them up.
